# serialization ‚Äî Go bebop + protobuf Cells ‚Äî Design

**Date:** 2026-07-16
**Status:** Approved

## Purpose

Extend the `serialization` focus area with two **Go-only** experiments, `bebop` and
`protobuf`, alongside the three existing Rust codecs (`sbe_gen`, `aeron_sbe`,
`bincode`). The cells measure the same thing the Rust cells measure ‚Äî encode/decode
latency and decode-time allocation for one ~500-byte SMR journal record ‚Äî so the
grid reads side-by-side: zero-copy SBE vs owned-decode codecs, now across two
languages.

This supersedes the "No Go or Java benchmark cell" non-goal in
`2026-07-13-serialization-design.md` for Go. Java remains out of scope.

## Experiments (two Go codecs)

| experiment | generator                                        | runtime dep                  | style                     |
|------------|--------------------------------------------------|------------------------------|---------------------------|
| `protobuf` | `protoc` + `protoc-gen-go` (pinned), regen script | `google.golang.org/protobuf` | eager owned decode        |
| `bebop`    | `github.com/200sc/bebop` Go generator, regen script | `github.com/200sc/bebop`   | eager owned decode, fixed layout |

- **`protobuf`** measures the **canonical** runtime, `google.golang.org/protobuf`
  (reflection-backed marshal/unmarshal) ‚Äî what "protobuf in Go" means to most
  readers. `vtprotobuf` is explicitly out of scope; it can be a later experiment row.
- **`bebop`** uses `github.com/200sc/bebop`, the de-facto Go implementation (the
  official betwixt-labs `bebopc` has no Go target). The benchmark times the **safe
  API** (`MarshalBebopTo`/`UnmarshalBebop`), not the unsafe fast path ‚Äî the way a
  production journal would use it, and fair against stock protobuf.

## Payload parity ‚Äî the cross-language fairness anchor

Both cells serialize the **same logical `JournalRecord`** as the Rust cells:
fixed header (five int64, two int32, event type, flags) plus an `entries` group
(default 4) of (int64 √ó3, int32, `command` byte blob, default 78 bytes) ‚Äî ~500
bytes encoded. Same env knobs: `SER_ENTRIES`, `SER_CMD_BYTES` (defaults 4 / 78),
`SER_WARMUP` (1000), `SER_ITERS` (100000).

Go ports the two deterministic primitives from Rust `serialization-common`:

- `build_record(index, entries, cmd_bytes)` ‚Äî splitmix64-seeded, index-derived,
  no RNG state, no wall clock.
- the FNV-1a-style order-sensitive `Checksum` fold (the full-materialization proof).

**Golden-checksum test:** a Go unit test asserts the ported builder + fold
reproduces **hardcoded checksum values generated by the Rust implementation**
for a handful of `(index, entries, cmd_bytes)` tuples (including the defaults).
This is the same role the golden snapshot test plays in `smr-collections`: green
means both languages benchmark the identical logical record. The golden values
are captured once from a small Rust snippet run against `serialization-common`
and pasted into the Go test with a comment saying how to regenerate them.

## Schemas

Two schema files, both expressing the `JournalRecord` layout:

**`journal.bop`** ‚Äî bebop `struct` (fixed layout, not `message`) for both
`JournalRecord` and `Entry`: `int64`/`int32` scalars, `byte` for
`eventType`/`flags`, `byte[] command`, `Entry[] entries`. Bebop structs match
SBE's fixed-block semantics naturally.

**`journal.proto`** (proto3, `package hiperf.journal`, `option go_package`):

- The seven id/timestamp/int fields use **`sfixed64`/`sfixed32`**, not default
  varints. Rationale: the record's ids and timestamps are full-width mixed bit
  patterns; proto3 varint `int64` encodes such values in ~10 bytes and negative
  `int32` at worst-case width. A real SMR journal schema with Aeron-style
  full-width ids would choose fixed-width fields; this also keeps
  `encoded_bytes` comparable instead of measuring varint pathology on synthetic
  data. (Decision confirmed 2026-07-16.)
- `event_type` and `flags` are `uint32` (proto3 has no 8-bit type); the Go cell
  narrows to/from `uint8` at the model boundary.
- `command` is `bytes`; `entries` is a `repeated Entry` message field.

Wire sizes will differ across codecs (SBE ‚âà 502 B; bebop and protobuf add their
own framing) ‚Äî `encoded_bytes` is a measured result, not a constant to force.

## Harness ‚Äî Go port of `run_journal`

Per the repo rule that each language's bench library owns Stats, env-config
parsing, the timed loop, and emission:

- **`go/internal/bench/serial.go`** ‚Äî `SerialConfig` + `LoadSerialConfig()`
  (`SER_*` knobs, same names/defaults as Rust, positive-integer validation that
  hard-errors like the other Go configs) and
  `RunJournal[R any](experiment string, cfg SerialConfig, build func(uint64) R, encode func(R, []byte) int, decode func([]byte) uint64)`
  ‚Äî generic over the record type, like Rust's `run_journal`, so `internal/bench`
  stays focus-neutral and never imports `serjournal`.
- **`go/internal/serjournal/`** ‚Äî the record model, `BuildRecord`, `Checksum`
  fold, and the generated codec packages (the `smrcoll` layout pattern).
- **`go/cmd/serialization-bebop/`**, **`go/cmd/serialization-protobuf/`** ‚Äî thin
  mains that load config and plug their encode/decode closures into `RunJournal`.

`RunJournal` mirrors `bench-common::serial::run_journal` phase for phase:

1. Pre-build all `warmup + iters` records (untimed).
2. Warmup encode, then timed encode: each record serialized into a **reused
   scratch buffer** ‚Äî protobuf via `proto.MarshalOptions{}.MarshalAppend(scratch[:0], ‚Ä¶)`,
   bebop via `MarshalBebopTo(scratch)` ‚Äî per-op ns samples.
3. Build one contiguous in-memory journal buffer with `(off, len)` frames.
4. Warmup decode, then timed decode: unmarshal + **fully materialize** ‚Äî fold
   every field of the decoded record through the Checksum in the canonical
   order, XOR into a sink consumed after the loop (`runtime.KeepAlive`/global
   sink) so reads cannot be elided.
5. Emit.

**Metrics** (identical names/units to the Rust cells): `encode_p50`, `encode_p99`
(int ns), `encode_mean` (float ns), `decode_p50`, `decode_p99`, `decode_mean`,
`encoded_bytes` (one record), `decode_alloc_bytes` (per decode).

**`decode_alloc_bytes` in Go:** `runtime.ReadMemStats` before and after the timed
replay loop; report `ŒîTotalAlloc / iters`. `TotalAlloc` is cumulative and
monotonic, so GC cannot deflate it ‚Äî the Go analog of Rust's counting allocator.
`ReadMemStats` runs exactly twice, outside any timed region. The loop runs on a
single goroutine with default GOGC; GC pauses landing in decode-latency tails
are honest Go replay pressure and are noted in RESULTS when relevant.

stdout carries only result lines; diagnostics go to stderr via `bench.Logf`/`Fatalf`.

## Codegen ‚Äî committed output, regen scripts

Generated code is **checked in** so bench hosts and CI need no `protoc` or
generator binaries (the `smr-collections/booksnap` precedent):

```
go/internal/serjournal/
  schema/journal.proto
  schema/journal.bop
  journalpb/    # protoc + protoc-gen-go output (committed)
  journalbop/   # 200sc/bebop generator output (committed)
  regen-journalpb.sh   # protoc --go_out=‚Ä¶ with a pinned protoc-gen-go version; records the protoc version used in the generated-file header
  regen-journalbop.sh  # go run the 200sc/bebop generator at the version pinned in go.mod
  serjournal.go        # JournalRecord model, BuildRecord, Checksum
  *_test.go
```

`go.mod` gains `google.golang.org/protobuf` and `github.com/200sc/bebop`
(runtime deps of the generated code). Regen scripts are dev-time only.

## Tests

- **Golden cross-language checksums** ‚Äî the fairness anchor (see above).
- **Round-trip per codec** ‚Äî `BuildRecord` ‚Üí encode ‚Üí decode+fold; checksum
  equals the direct fold of the source record.
- **Size sanity** ‚Äî at default knobs each codec's `encoded_bytes` lands in a
  band around ~500 B (loose enough for per-codec framing differences), guarding
  the knob defaults.
- **Builder determinism** ‚Äî same index twice ‚Üí identical record; different
  indices differ.

`go build ./... && go vet ./... && go test ./...` stays green.
